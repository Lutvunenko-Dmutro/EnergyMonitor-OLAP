# РОЗДІЛ 2. ПОСТАНОВКА ЗАВДАННЯ ТА ВИМОГИ ДО СИСТЕМИ

## 2.1. Формулювання задачі кваліфікаційного проєктування
Основною задачею цього кваліфікаційного проєктування є створення комплексної інтелектуальної SaaS-платформи EnergyMonitor-OLAP, призначеної для моніторингу, симуляції фізичних станів та предиктивного аналізу часових рядів енергоспоживання в інфраструктурі Smart City. Система повинна забезпечувати перехід від реактивного управління енергомережами до проактивного (Predictive Maintenance) шляхом використання рекурентних нейронних мереж (LSTM) та архітектури багатовимірного аналізу даних (OLAP).

## 2.2. Вхідна та вихідна інформація системи
Для забезпечення адекватної роботи предиктивних моделей та механізму цифрового двійника, система повинна обробляти гібридний потік даних, що поділяється на історичні ретроспективи та реальну телеметрію.

**Вхідна інформація включає:**
1.  **Історичну базу даних**: Набір даних PJM Interconnection (Dayton Hourly), що містить верифіковані погодинні записи енергоспоживання у мегаватах (МВт).
2.  **Динамічну телеметрію (Real-time)**: Дані, що генеруються програмним цифровим двійником із частотою оновлення 15–60 хвилин:
    *   Енергетичні показники (поточне фактичне навантаження, напруга, частота струму).
    *   Метеорологічні дані (температура повітря, відносна вологість, тип атмосферних опадів).
    *   Діагностичні показники технічного стану обладнання (температура масла трансформаторів, концентрація розчиненого водню H2, рівень вібрації).
3.  **Часові детермінанти**: Повна циклічна розгортка часу (година доби, день тижня, маркування вихідних та святкових днів) для коректного виявлення патернів сезонності моделлю машинного навчання.

**Вихідні дані та засоби візуалізації:**
Для ефективної взаємодії оператора з системою, користувацький інтерфейс повинен бути реалізований у форматі аналітичної панелі (Dashboard), що забезпечує виведення такої інформації:
*   **Предиктивна візуалізація**: Інтерактивні графіки з відображенням фактичного навантаження та лінії прогнозу на горизонт 24–48 годин із розрахованим «коридором довіри» (Confidence Interval).
*   **Геоінформаційний шар**: Інтерактивна топологічна мапа, де вузли мережі (підстанції) відображаються у вигляді кольорових маркерів, що динамічно змінюють стан (від нормального функціонування до критичного перевантаження).
*   **Система сповіщень (Alerting)**: Журнал критичних подій та аномалій (стрибки навантаження, температурна деградація обладнання) з відповідними часовими мітками.
*   **Статистичний аудит**: Аналітичні зведення щодо точності роботи ML-ядра, що включають метрики похибки (MAPE, RMSE, R²), а також результати статистичних тестів на нормальність розподілу помилок (тест Шапіро-Вілка).

## 2.3. Вимоги до технічного та програмного забезпечення
Спроєктований програмний комплекс має функціонувати у хмарному середовищі з дотриманням принципів безперервного розгортання та контейнеризації.

**Вимоги до серверної частини:**
*   **Платформа розгортання**: Хмарний PaaS-провайдер (наприклад, Render) або виділений сервер під управлінням ОС сімейства Linux.
*   **Апаратні ресурси**: Мінімальний обсяг оперативної пам'яті (RAM) становить 1 ГБ (через специфіку завантаження вагових коефіцієнтів моделей TensorFlow), процесор — 1-core CPU.
*   **Програмне середовище**: Обов'язкова контейнеризація за допомогою Docker (на базі образу Debian/Python 3.11-slim або вище).
*   **База даних**: Реляційна система керування базами даних PostgreSQL версії 15 або вище (рекомендовано 16, з використанням хмарної інфраструктури Neon Cloud для оптимізації OLAP-запитів).

**Вимоги до клієнтської частини:**
*   Сучасний веб-браузер (Google Chrome, Mozilla Firefox, Safari) з увімкненою підтримкою JavaScript та WebGL для плавного рендерингу складних графічних компонентів (Plotly, Folium).
*   Мінімальна роздільна здатність екрана: 1280x720 пікселів для повнофункціонального відображення аналітичних графіків та карт.
*   Наявність стабільного інтернет-з'єднання зі швидкістю не менше 1 Мбіт/с для отримання безперервного потоку телеметричних даних.

## 2.4. Етапи проєктування системи
Відповідно до поставлених завдань, розробка системи передбачає таку черговість етапів, які можуть мати ітеративний характер (зокрема на етапах підготовки даних та проєктування БД):
1.  Збір, очищення та попереднє оброблення історичних даних датасету PJM.
2.  Розробка схеми бази даних PostgreSQL та розгортання цифрового двійника для симуляції сенсорів.
3.  Проєктування, навчання та валідація предиктивної моделі глибокого навчання (LSTM v3).
4.  Створення інтерактивного користувацького інтерфейсу на базі фреймворку Streamlit.
5.  Контейнеризація застосунку та налаштування CI/CD конвеєра для деплою у хмарне середовище.

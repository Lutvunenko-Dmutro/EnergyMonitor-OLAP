# 🛡️ Звіт про безпеку (Security Audit Report)

Проект **Energy Monitor ULTIMATE** розроблений із пріоритетом на безпеку даних та стійкість до атак. У цьому документі описано стратегію захисту та результати аудиту.

---

## 🏗️ Стратегія захисту (Defense in Depth)

Система використовує багатошаровий підхід до безпеки:

1.  **Транспортний рівень (SSL/TLS)**: Весь трафік до БД Neon Cloud шифрується (режим `require`).
2.  **Прикладний рівень (Validation)**: Сувора валідація всіх вхідних параметрів через регулярні вирази.
3.  **Рівень БД (Parametrization)**: Виключення динамічної збірки SQL-запитів.
4.  **Рівень операцій (Secrets)**: Відсутність захардкоджених паролів у коді.

---

## 🔒 Захист від SQL Injection

Це найбільш критична вразливість для OLAP-систем. У проекті реалізовано:

### 1. Параметризовані запити
Всі запити до БД використовують токени (`:param`), що робить ін'єкцію коду технічно неможливою на рівні драйвера.
```python
# src/core/database.py
run_query("SELECT * FROM LoadMeasurements WHERE id = :id", {"id": 123})
```

### 2. Whitelisting (Білі списки)
Для динамічних частин запиту (наприклад, назв колонок), які не можна параметризувати, використовуються валідатори з `utils/validators.py`:
```python
VALID_COLUMNS = {"actual_load_mw", "health_score", "timestamp"}
if col not in VALID_COLUMNS:
    raise ValueError("Illegal column access detected!")
```

## 🩹 Захист від XSS та маніпуляцій UI

*   **Streamlit Sanitization**: Весь текст, що відображається, автоматично екранується Streamlit. 
*   **Markdown safety**: `unsafe_allow_html=True` використовується лише у перевірених футерах, де немає вводу користувача.

---

## 💂 Управління секретами та доступом

*   **Environment Variables**: Всі чутливі дані (паролі, хости) завантажуються виключно через `os.getenv`.
*   **Git Safety**: Файл `.env` занесений до `.gitignore`.
*   **GitHub Secrets**: Для CI/CD пайплайнів використовуються зашифровані секрети GitHub.
*   **Redacted Logging**: Логер проекту автоматично перехоплює credentials та не виводить їх у консоль.

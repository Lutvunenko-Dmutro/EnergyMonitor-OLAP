    • [x] [Task] Обґрунтування актуальності та постановка завдань дослідження
      • [x] Абзац 1 (Актуальність): Проблематика енергомереж, воєнний стан та необхідність переходу від простого спостереження за мережею до завчасного прогнозування її стану.
      • [x] Абзац 2 (Технологічна база): Використання LSTM v3 (9 факторів), концепції Digital Twin (теплова модель трансформатора) та хмарної інфраструктури Neon/Render.
      • [x] Методологічний блок: Об'єкт (процеси енергоспоживання), Предмет (методи LSTM та OLAP-архітектура), Мета (SaaS-платформа EnergyMonitor).
      • [x] Перелік завдань: 5 етапів – від аналізу методів до верифікації точності (MAPE < 3.1%).
      • [x] Практичне значення: Впровадження Health Score діагностики та прогнозування на 48 годин наперед (Sliding Window).
      • [x] Опис структури: Огляд 3-х розділів, що охоплюють теорію, вимоги та повний цикл реалізації (включаючи CI/CD та Docker).
      • [x] «Золоте правило» цитування: Додано плейсхолдери [*] для Smart City, Digital Twin, LSTM та MAPE згідно з вимогами керівника.
      • [x] Технічні нюанси: Усунено "стиль ChatGPT" (жирні списки), розшифровано MAPE, виправлено орфографію (патерн) та додано академічну пунктуацію.

    🚀 РОЗДІЛ 1. ОГЛЯД ЛІТЕРАТУРИ ТА АНАЛІЗ ПРЕДМЕТНОЇ ОБЛАСТІ

    • [x] [Task] 1.1. Інтелектуальні енергосистеми та концепція Smart City
      • [x] 1.1.1. Виклики та досвід:
        • Абзац 1: Енергобезпека України в умовах війни, дефіцит потужностей (30%) та необхідність предиктивного аналізу.
        • Абзац 2: Використання IoT-датчиків та AMI-лічильників (15-60 хв) як фундаменту для прогнозів.
        • Абзац 3: Світовий досвід (Сінгапур, Барселона) та локальне дослідження погодних факторів Києва [39].
      • [x] 1.1.2. Smart Grid:
        • Абзац 1: Визначення Smart Grid та переваги двостороннього обміну енергією та даними.
        • Абзац 2: Технічні компоненти: інтелектуальний облік (AMI), пристрої PMU та системи ESS.
        • Абзац 3: Порівняння з традиційними мережами та вирішення проблеми "вечірнього піку" (duck curve).
      • [x] 1.1.3. Digital Twin:
        • Абзац 1: Визначення за ISO 23247 та IEEE 1547. Моделювання теплових процесів у трансформаторі.
        • Абзац 2: Розрахунок H₂ та Health Score (IEEE C57.91) для предиктивного обслуговування.
    • [x] [Task] 1.2. Математичний апарат глибокого навчання (LSTM)
      • [x] 1.2.1. Часові ряди:
        • Абзац 1: Математична декомпозиція навантаження: тренд, добова/тижнева сезонність та шум.
      • [x] 1.2.2. Математичний апарат LSTM:
        • Абзац 1: Обґрунтування вибору LSTM для нелінійних рядів та механізм гейтів.
        • Абзац 2: Повний набір рівнянь: Forget Gate, Input Gate, Cell State update (\(C_t\)), Output Gate та фінальне значення (\(h_t\)).
        • Абзац 3: Опис переваг – прогнозування без ручного створення сотень ознак (feature engineering).
        • Абзац 4: Оптимізація навчання: використання Adam та Huber Loss для захисту від аномалій.
    • [x] [Task] 1.3. Технології Big Data та OLAP
      • [x] 1.3.1. Обґрунтування:
        • Абзац 1: Порівняння OLTP та OLAP. Вибір PostgreSQL (Neon Cloud) для агрегаційних запитів (AVG, SUM) у реальному часі.
    • [x] [Task] 1.4. Порівняльний аналіз методів прогнозування
      • [x] 1.4.1. Архітектури:
        • Абзац 1: Порівняння RNN та LSTM (проблема зникаючого градієнта).
      • [x] 1.4.2. ARIMA:
        • Абзац 1: Обмеження класичної статистики на нестаціонарних воєнних рядах.
      • [x] 1.4.3. Класичне ML:
        • Абзац 1: Проблема ручного створення lag-ознак в XGBoost/Random Forest.
      • [x] 1.4.4. Глибоке навчання (LSTM):
        • Абзац 1: Обґрунтування LSTM та порівняння з Transformers (баланс швидкості та точності).
        • Абзац 2: Підсумок аналізу: вибір LSTM як ядра системи EnergyMonitor-OLAP.
    • [x] [Task] Графіка та оформлення:
      • [x] Mermaid-схеми (GitHub) та PNG (Word). Підписи «Джерело: розроблено автором» згідно з вимогами ДСТУ.
